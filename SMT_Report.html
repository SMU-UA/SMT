System Monitoring Tool

A persistent background server that captures every test transition in real time — bridging hardware-in-the-loop testing with live Modbus telemetry.

Architecture Report — March 2026

1 The Core Idea

Traditional HIL testing treats the test runner and the monitoring system as two completely separate worlds. The test script runs scenarios, saves .mat files, and moves on — while the engineer manually correlates waveforms with fault codes after the fact.

SMT flips this model: a lightweight HTTP server runs persistently in the background, and every test scenario automatically announces itself to the server as it starts and ends. The server broadcasts these events via Server-Sent Events (SSE) to a live dashboard that is simultaneously sniffing the Modbus RTU bus. The result is a perfectly time-aligned record of what the test was doing and what the hardware was reporting.

High-Level Architecture

2 Three-State Scenario Protocol

The communication between test scripts and the server uses a simple but powerful three-state model that elegantly handles the gap between scenarios without losing data or triggering premature saves.

State Machine

The "paused" state is the key innovation: when a test scenario ends (post_cbk), the dashboard stops capturing CSV rows but keeps the log session open. When the next scenario begins (pre_cbk), capture resumes seamlessly. This prevents the log from being cluttered with idle data between tests, while avoiding premature auto-saves that would fragment the recording.

3 Why a Persistent Background Server?

The most interesting architectural decision in SMT is that the server runs independently of the tests. It starts before any test begins and stays alive after all tests finish. This decoupled design has several powerful implications:

Always-On Capture

The dashboard and serial connection stay alive across test sessions. Re-run a failing test without restarting the monitoring stack.

Multi-Client Observation

Multiple browser tabs can connect to the SSE stream. One engineer runs tests while another watches the dashboard remotely.

Crash Detection

Heartbeat monitoring detects if the test runner crashes unexpectedly and auto-saves the log — no data lost.

Zero Config UI

No install needed — plain HTML served by the Python server. Works in any modern browser with Web Serial support.

4 The Vision: A Universal Test Notification API

Today, only SystemLevel_Scenarios.py talks to the SMT server. But the architecture is inherently open — any script, in any language, on any machine can notify the server with a simple HTTP POST. This opens the door to a universal test notification API.

Future: Open API Architecture

The API contract is intentionally minimal — a single JSON POST is all it takes to participate:

// Any language, any platform — just HTTP POST

POST http://localhost:8780/scenario
Content-Type: application/json

{
  "label": "MyTest_Scenario_1",
  "running": true  // true | "paused" | false
}

// MATLAB example:
webwrite('http://localhost:8780/scenario', ...
    struct('label','DAB_Test_1', 'running',true), ...
    weboptions('MediaType','application/json'))

// curl example:
curl -X POST http://localhost:8780/scenario \
  -H "Content-Type: application/json" \
  -d '{"label":"MyTest_1","running":true}'

This means anyone writing a test scenario — in Python, MATLAB, LabVIEW, C#, or even a shell script — can instantly integrate with the SMT monitoring pipeline. No SDK, no library, no special protocol. Just a two-field JSON payload over HTTP.

5 Roadmap

Phase 1 — Completed

Core server with SSE, three-state protocol, Modbus dashboard, heartbeat crash detection, CSV data logging with scenario labels.

Phase 2 — In Progress

MATLAB report generator with GUI (SMT_Report.m): combines HIL .mat data with Modbus CSV into IEEE-style PDF reports. Includes built-in CSV splitting — eliminates the separate Python tool.

Phase 3 — Future

Publish the API specification so any test framework can notify the server. Add optional server-side CSV save (bypass browser Downloads). Webhook integration for pass/fail notifications.

Phase 4 — Vision

Auto-trigger MATLAB report generation when a test suite completes. Historical test run database. CI/CD integration for automated HIL regression testing with monitoring.
